  <action>
Create reporters/rspec/lib/tdd_guard_rspec/formatter.rb implementing the TDD Guard formatter.

The formatter must:

1. **Inherit from RSpec::Core::Formatters::BaseFormatter** and register for notifications: `example_passed`, `example_failed`, `example_pending`, `dump_summary`, `close`.

2. **Collect test results** in an array. For each example notification:
   - Extract `name` from `example.description` (the innermost describe/it text, e.g., "should add numbers correctly")
   - Extract `fullName` from `example.full_description` (the full hierarchical name, e.g., "Calculator should add numbers correctly")
   - Set `state` to "passed", "failed", or "skipped" (map pending to skipped)
   - For failed tests, extract errors array with:
     - `message`: full formatted failure message from `notification.message_lines.join("\n")` or `example.execution_result.exception.message` combined with the formatted backtrace
     - `expected`: from `example.execution_result.exception` if it responds to `expected` (RSpec::Expectations::ExpectationNotMet has this)
     - `actual`: from `example.execution_result.exception` if it responds to `actual`

3. **Handle syntax/load errors** by registering for the `dump_summary` notification. In `dump_summary`, check `notification.errors_outside_of_examples_count` and iterate `summary.examples` to find load errors. Alternatively, use a simpler approach: override `example_failed` to handle all failures including load-time. The key insight is that RSpec reports load errors differently -- they appear in `dump_summary` via `notification.failed_examples` or through `RSpec.world.wants_to_quit`.

   The most reliable approach for load/syntax errors: Register for `dump_summary`. Check `summary_notification.errors_outside_of_examples_count > 0`. If so, create a synthetic test entry with:
   - `name`: a descriptive name like "load_error" or derived from the file path
   - `fullName`: the file path or error source
   - `state`: "failed"
   - `errors`: array with the error message

   Actually, the cleanest pattern (matching pytest's `pytest_collectreport`): In RSpec, load errors prevent examples from running. They surface as exceptions during `load`. To catch these, wrap the formatter to also handle the `dump_summary` notification where `notification.errors_outside_of_examples_count` is available. The actual error details are in the `output` stream or can be captured from `$stderr`.

   A more reliable approach: Simply check after all examples have run. If there are no test results but the run had errors, create a synthetic entry. In `dump_summary`, inspect if `notification.failed_examples` is empty but `notification.errors_outside_of_examples_count > 0` -- this means load errors occurred.

   For load errors, RSpec 3.x surfaces them via the `message` notification. Register for the `message` notification (`RSpec::Core::Formatters.register self, :message`). In `message(notification)`, check if `notification.message` contains error text (load errors produce messages like "An error occurred while loading spec_file.rb"). Store these messages. Then in `dump_summary`, if `notification.errors_outside_of_examples_count > 0`, create synthetic test entries from the stored messages.

4. **Group tests by module** (spec file path). Use `example.metadata[:file_path]` to get the relative file path. Strip leading `./` if present. Group all tests from the same file into the same `testModule` entry with that path as `moduleId`.

5. **Set overall reason** field: "passed" if all tests passed and no errors, "failed" if any test failed or any load error occurred.

6. **Write JSON output** in the `close` callback. Build the output hash:
   ```ruby
   {
     "testModules" => [...],
     "unhandledErrors" => [],
     "reason" => "passed" | "failed"
   }
   ```

   Write to `.claude/tdd-guard/data/test.json` relative to the project root. The project root defaults to the current working directory (Dir.pwd). Create the directory with `FileUtils.mkdir_p` before writing.

   Use `JSON.generate(output)` (not `JSON.pretty_generate`) for compact output, or use `JSON.pretty_generate` for readability -- either works, the integration tests parse JSON regardless.

Important implementation notes:
- Use `require "json"` and `require "fileutils"` at the top of the file.
- The formatter is used via `--format TddGuardRspec::Formatter --require tdd_guard_rspec`. The class must be accessible by that fully qualified name.
- Do NOT set `reason` to `nil` or omit it -- always set it to "passed" or "failed" (matching all other reporters).
- For the `message` notification: the notification object has a `message` method returning the string. Load error messages typically start with "An error occurred while loading".
- Store the load error file path by parsing the message: extract the file path from "An error occurred while loading ./path/to/spec.rb" and use it as the moduleId.
