## Directory Purposes

**src/cli/:**
- Purpose: Hook entry point and context aggregation
- Contains: tdd-guard.ts (CLI bin), buildContext.ts (parallel data loading)
- Key files: `src/cli/tdd-guard.ts` (stdin handler), `src/cli/buildContext.ts` (Storage → Context)

**src/config/:**
- Purpose: Configuration resolution from env/options
- Contains: Config class with defaults, validation, path resolution
- Key files: `src/config/Config.ts`

**src/contracts/:**
- Purpose: Type definitions and runtime validation schemas
- Contains: TypeScript types (interfaces), Zod schemas (runtime validation)
- Key files:
  - Types: `src/contracts/types/Context.ts`, `src/contracts/types/ModelClient.ts`, `src/contracts/types/ValidationResult.ts`
  - Schemas: `src/contracts/schemas/toolSchemas.ts` (Hook/Operation), `src/contracts/schemas/reporterSchemas.ts` (Test results), `src/contracts/schemas/pytestSchemas.ts`, `src/contracts/schemas/lintSchemas.ts`

**src/guard/:**
- Purpose: Guard enable/disable state and file ignore patterns
- Contains: GuardManager with minimatch file pattern filtering
- Key files: `src/guard/GuardManager.ts`

**src/hooks/:**
- Purpose: Hook event processing and routing
- Contains: Event handlers (Session, User commands, Post-tool lint), event extraction
- Key files:
  - `src/hooks/processHookData.ts` (main routing logic)
  - `src/hooks/HookEvents.ts` (operation extraction)
  - `src/hooks/sessionHandler.ts` (SessionStart)
  - `src/hooks/userPromptHandler.ts` (User commands)
  - `src/hooks/postToolLint.ts` (Lint notifications)

**src/linters/:**
- Purpose: Language-specific code quality checking
- Contains: Linter interface, ESLint and golangci-lint implementations
- Key files:
  - `src/linters/Linter.ts` (interface)
  - `src/linters/eslint/ESLint.ts` (JavaScript/TypeScript)
  - `src/linters/golangci/GolangciLint.ts` (Go)

**src/processors/:**
- Purpose: Transform raw data (test results, lint output) into readable context
- Contains: TestResultsProcessor (Vitest/pytest), lintProcessor (file grouping)
- Key files:
  - `src/processors/testResults/TestResultsProcessor.ts` (handles double-encoded JSON, formats failures)
  - `src/processors/lintProcessor.ts` (groups issues by file)

**src/providers/:**
- Purpose: Factory implementations for dependency injection
- Contains: ModelClientProvider (SDK/API/CLI), LinterProvider (language selection)
- Key files:
  - `src/providers/ModelClientProvider.ts` (model selection by Config.validationClient)
  - `src/providers/LinterProvider.ts` (linter selection by Config.linterType)

**src/storage/:**
- Purpose: Abstract persistence for transient hook data
- Contains: Storage interface, FileStorage (filesystem), MemoryStorage (in-memory)
- Key files:
  - `src/storage/Storage.ts` (interface)
  - `src/storage/FileStorage.ts` (filesystem using `.claude/tdd-guard/data/`)
  - `src/storage/MemoryStorage.ts` (for testing)

**src/validation/:**
- Purpose: AI-based TDD rule validation
- Contains: Validator orchestration, context generation, model clients, prompts
- Subdirectories:
  - `src/validation/validator.ts` - Main validator, handles model response parsing (JSON extraction strategies)
  - `src/validation/context/` - Dynamic context building from storage data
  - `src/validation/models/` - Model client implementations (ClaudeAgentSdk, AnthropicApi, ClaudeCli)
  - `src/validation/prompts/` - Modular prompt sections (system, rules, file types, operations, tools)

**test/:**
- Purpose: Unit, integration, and fixture testing
- Contains: Tests for all src modules, test utilities, sample hook data
- Subdirectories:
  - `test/hooks/` - Tests for hook processing
  - `test/integration/` - Multi-layer integration tests
  - `test/utils/` - Test helpers and factories
  - `test/artifacts/` - Sample hook events and reporter output

**reporters/:**
- Purpose: Language-specific test result reporters for integration with hook system
- Contains: Separate npm/pip/gem/go packages for each test framework
- Key:
  - `reporters/vitest/`, `reporters/jest/`, `reporters/storybook/` - npm workspaces
  - `reporters/pytest/` - pip package
  - `reporters/rspec/`, `reporters/phpunit/` - gem/composer packages
  - `reporters/go/` - go package
  - `reporters/test/` - shared test artifacts and factories

## Key File Locations

**Entry Points:**
- `src/cli/tdd-guard.ts` - CLI binary (bin: tdd-guard), receives hook JSON on stdin
- `src/index.ts` - Package entry point (exports Storage, Config, types, schemas)
- `reporters/*/src/index.ts` - Reporter entry points (framework-specific)

**Configuration:**
- `src/config/Config.ts` - Resolves env/options to configuration
- `.claude/hooks/` - Claude Code hook configuration (hooks context)
- `package.json` - npm bin definition, workspaces, scripts

**Core Logic:**
- `src/hooks/processHookData.ts` - Main hook routing and event dispatch
- `src/cli/buildContext.ts` - Parallel data loading from storage
- `src/validation/validator.ts` - AI validation with response parsing
- `src/validation/context/context.ts` - Dynamic prompt assembly

**Data Persistence:**
- `src/storage/FileStorage.ts` - Filesystem backend at `.claude/tdd-guard/data/`
  - `test.json` - Test results from reporter
  - `modifications.json` - File operation (Edit/Write/MultiEdit)
  - `todos.json` - Todo operations
  - `lint.json` - Lint results from linter
  - `config.json` - Guard state (enabled/disabled, ignore patterns)
  - `instructions.md` - TDD rules (customizable)

**Type Contracts:**
- `src/contracts/types/Context.ts` - Validation context shape
- `src/contracts/types/ValidationResult.ts` - Model response shape
- `src/contracts/types/ModelClient.ts` - Model client interface
- `src/contracts/schemas/toolSchemas.ts` - Hook and operation validation

**Testing:**
- `test/utils/factories/` - Test data generators
- `reporters/test/factories/` - Reporter-specific factories
- `test/artifacts/` - Sample hook events, test outputs, lint results
- `vitest.config.ts` - Test runner configuration with multiple projects

## Naming Conventions

**Files:**
- `[Name].ts` - Source files
- `[Name].test.ts` - Unit tests (co-located with implementation)
- `[Name]Provider.ts` - Factory classes (ModelClientProvider, LinterProvider)
- `[Name]Manager.ts` - State/config managers (GuardManager)
- `[Name]Schema.ts` - Zod schema definitions (in schemas/ directory)
- `[Name]Handler.ts` - Event/request handlers (SessionHandler, UserPromptHandler)
- `index.ts` - Public API exports for modules

**Directories:**
- `schemas/` - Zod runtime validation schemas
- `types/` - TypeScript interface definitions
- `models/` - Model client implementations
- `operations/` - Operation-specific prompt sections
- `tools/` - Tool output prompt sections (test, lint, todos)
- `testResults/` - Test processing utilities
- `eslint/`, `golangci/` - Language-specific implementations

**Classes:**
- PascalCase (Config, Storage, FileStorage, TestResultsProcessor)
- Suffix patterns: Provider, Manager, Handler, Schema

**Functions:**
- camelCase (buildContext, processHookData, generateDynamicContext, formatLintDataForContext)
- Prefix patterns: format*, process*, generate*, extract*, should*, is* (predicates)

**Exports:**
- Named exports for classes, interfaces, factories
- Default exports: none (use named exports for consistency)
- Re-exports from index.ts for public API

## Where to Add New Code

**New Feature (TDD validation rule or enforcement):**
- Modify: `src/validation/prompts/rules.ts` (add rule to system prompt)
- Modify: `src/validation/context/context.ts` (if new context data needed)
- Modify: `src/validation/validator.ts` (if new response parsing logic)
- Tests: `test/integration/` or `test/validation/` (integration tests with mocked model)

**New Hook Event Type:**
- Modify: `src/contracts/schemas/toolSchemas.ts` (extend HookData schema)
- Add handler in: `src/hooks/processHookData.ts` (new route in process logic)
- Add handler class in: `src/hooks/` (e.g., NewEventHandler.ts)
- Tests: `test/hooks/`

**New Model Client (e.g., Bedrock, local LLM):**
- Add: `src/validation/models/NewClient.ts` (implement IModelClient interface)
- Modify: `src/providers/ModelClientProvider.ts` (add case in switch statement)
- Add env var handling in: `src/config/Config.ts` (if needed)
- Tests: `src/validation/models/NewClient.test.ts`

**New Linter (e.g., Pylint, Clippy):**
- Add: `src/linters/[language]/NewLinter.ts` (extend Linter base class)
- Modify: `src/providers/LinterProvider.ts` (add case in factory)
- Add schema in: `src/contracts/schemas/lintSchemas.ts` (if new result format)
- Tests: `src/linters/[language]/NewLinter.test.ts`

**New Test Reporter (e.g., Playwright, Cypress):**
- Add: `reporters/[framework]/` (new workspace directory)
- Implement: Save test results to `.claude/tdd-guard/data/test.json` format (see `src/contracts/schemas/reporterSchemas.ts`)
- Add to: `package.json` workspaces array
- Tests: `reporters/test/` or `reporters/[framework]/tests/`

**Utilities & Helpers:**
- Shared helpers: `src/processors/` (data transformation)
- Test factories: `test/utils/factories/` (test data generators)
- Constants: Define near usage or in `src/contracts/` if schema-related

## Special Directories

**`.claude/tdd-guard/data/`:**
- Purpose: Runtime hook data persistence
- Generated: Yes (auto-created by FileStorage on first write)
- Committed: No (in .gitignore)
- Contents:
  - `test.json` - Current test run results (JSON from reporter)
  - `modifications.json` - Latest file operation
  - `todos.json` - Latest todo operation
  - `lint.json` - Current lint results
  - `config.json` - Guard state (can be manually edited)
  - `instructions.md` - Custom TDD rules (can be manually edited)

**`reporters/test/artifacts/`:**
- Purpose: Sample test output files for reporter testing
- Generated: No
- Committed: Yes
- Contents: Example JSON outputs from each test framework

**`test/artifacts/`:**
- Purpose: Sample hook events for hook processing tests
- Generated: No
- Committed: Yes
- Contents: Example hook JSON payloads from Claude Code

**`docs/adr/`:**
- Purpose: Architecture decision records
- Generated: No
- Committed: Yes
- Format: Markdown ADRs explaining key design decisions

**`node_modules/`:**
- Purpose: Installed dependencies (npm workspaces)
- Generated: Yes (npm install)
- Committed: No (in .gitignore)
